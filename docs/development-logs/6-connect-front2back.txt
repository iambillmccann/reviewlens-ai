DECISIONS

1. Endpoint Path: /api/me (unversioned)
   - Canonical endpoint for app-shell identity
   - Separate router from versioned /api/v1 endpoints
   - Allows future API consistency work without breaking this flow

2. Payload Source: DB-backed app user profile
   - NOT raw Clerk token claims
   - Enables future profile customization, sync policies, etc.
   - Bootstrap flow creates user/workspace on first auth request

3. Compatibility Strategy
   - /api/v1/auth/me remains for backwards compatibility and debugging
   - Frontend switched to /api/me (canonical)
   - No breaking changes to existing tests or endpoints

4. UUID Serialization
   - AppUserProfile.id uses Python uuid.UUID type
   - Pydantic auto-serializes to string in JSON response
   - Frontend receives id as string UUID

FURTHER CONSIDERATIONS

1. Success Payload Envelope
   - Current codebase returns plain success objects: { id, email, display_name, avatar_url }
   - API conventions doc suggests: { "data": {}, "meta": {} } envelope
   - Recommendation: Keep current style for this step
   - Action: Treat envelope migration as separate API consistency task (not blocking)

2. Endpoint Versioning Strategy
   - /api/me is intentionally unversioned for app-shell identity
   - Versioned endpoints continue for broader API resources (/api/v1/*)
   - Recommendation: Document this pattern as canonical convenience endpoint approach

3. Avatar Field Handling
   - Currently trusts Clerk picture claim on first bootstrap
   - If both token image and persisted avatar differ, persisted DB value is authoritative thereafter
   - Future work: Define explicit profile-sync/update policy for avatar refresh

4. Error Handling
   - Missing bearer token → 401 "Missing bearer token"
   - Invalid/expired token → 401 "Invalid or expired bearer token"
   - Server auth misconfiguration → 503 "Server auth verification is not configured"
   - Database connection issues → Error propagated to response

DEBUGGING & RESOLUTION (May 11, 2026)

ISSUE: Profile Fields Not Persisting
  Symptom:
    - After fresh login, frontend showed "User" and "Not set" fallback values
    - DB query confirmed email/display_name/avatar_url all NULL
    - All unit tests passing (20/20)
    - CORS headers correct
    - No browser console errors

  Root Cause Analysis:
    - Implemented Clerk profile fallback service to enrich missing JWT claims
    - Service calls Clerk API endpoint: GET https://api.clerk.com/v1/users/{user_id}
    - Request was returning 403 Forbidden
    - Cloudflare error code 1010 in response body

  Investigation Steps:
    1. Added comprehensive debug logging to /api/me endpoint
    2. Added logging to _resolve_identity_profile() function
    3. Added logging to clerk_profile service functions
    4. Configured logging in app/main.py at DEBUG level
    5. Restarted backend completely (hot reload was insufficient)
    6. Triggered fresh login from frontend
    7. Observed backend logs showing Clerk API HTTP 403
    8. Tested Clerk API call directly with Python urllib
    9. Confirmed error: Cloudflare error code 1010 (IP/bot block)

  Root Cause: Cloudflare Protection
    - Cloudflare protects api.clerk.com
    - Python urllib default request has no User-Agent header
    - Cloudflare blocks requests without User-Agent as potential bot traffic
    - Error 1010 = "Client IP is restricted" / bot detection
    - Issue affects all urllib requests to Clerk management API

SOLUTION

  Implementation:
    - Updated apps/api/app/services/clerk_profile.py fetch_clerk_user_profile()
    - Added three headers to Request object:
      * Authorization: Bearer {secret_key}  (existing)
      * User-Agent: cornerstone-api/0.1.0  (new)
      * Accept: application/json           (new)

  Code Change:
    ```python
    request = Request(
        url=f"https://api.clerk.com/v1/users/{user_id}",
        headers={
            "Authorization": f"Bearer {secret_key}",
            "User-Agent": "cornerstone-api/0.1.0",
            "Accept": "application/json",
        },
        method="GET",
    )
    ```

  Result:
    - Clerk API now returns 200 OK
    - User profile successfully fetched: first_name, last_name, email_addresses[], image_url
    - Profile fields extracted and backfilled to DB on next /api/me call
    - Frontend now displays real user data: "Bill McCann", "billfmccann@hotmail.com"

LESSONS LEARNED

  1. Third-party API Protection
     - Many CDN-protected APIs (Cloudflare, WAF) block requests without User-Agent
     - urllib's default headers are insufficient for production APIs
     - Always test third-party integrations with explicit headers

  2. Debug Logging Strategy
     - Comprehensive logging at decision points helps identify execution flow
     - Prefix logs with [COMPONENT] for easy filtering in output stream
     - Include both happy path logs and fallback path logs
     - Include error details (not just errors: true/false)

  3. Hot Reload Limitations
     - Hot reload works for most code changes
     - Logging configuration at module import time may require full restart
     - When in doubt during active debugging, do a complete restart

  4. Separation of Concerns
     - JWT enrichment logic separate from Clerk API calls
     - Profile extraction logic separate from HTTP transport
     - Made it easier to isolate and fix the HTTP layer issue
