
MILESTONE: Infrastructure as Code Refactor (IaC / AWS / Terraform)
Date: May 18–20, 2026
Status: Step 1 Complete (AWS OIDC trust)

GOAL
Move Cornerstone from platform-pull deployment (Cloudflare/Railway auto-deploy on push) to explicit IaC-driven
deployments using Terraform-managed AWS infrastructure, starting with AWS as the first cloud provider adapter.

DECISION RATIONALE
- Cornerstone is a template, not a production product. Deployment must be reproducible, testable, and easy to tear down.
- Platform-pull deployments (auto-deploy on push) are designed for continuously running SaaS products, not templates.
- Explicit IaC (Terraform + GitHub Actions) gives Cornerstone operators full visibility and control.
- Provider-adapter architecture enables future GCP/Azure support without changing application code.
- Pause/resume capability (App Runner + RDS) allows cost-effective dev environment validation.

ARCHITECTURE HIGHLIGHTS
- Frontend: S3 + CloudFront (static site, HTTPS, SPA routing)
- Backend: AWS App Runner (managed container, stable URL, pause/resume)
- Database: RDS PostgreSQL 16 (t4g.micro, private access from backend only)
- Networking: VPC with public + private subnets, NAT egress for backend, private RDS
- Secrets: created by bootstrap script, referenced by ARN in Terraform (no secrets in .tfstate)
- Deployment: GitHub Actions orchestration; OIDC auth (no long-lived keys)
- Teardown: two-tier (pause = reversible, destroy = nuke)

INFRASTRUCTURE DECISIONS
- AWS provider selected as initial adapter
- python:3.12-slim Docker base image (matches local dev environment)
- GitHub OIDC authentication (no long-lived AWS keys in GitHub secrets)
- Secrets Manager created by bootstrap script (not managed by Terraform)
- No custom domain initially (native App Runner + CloudFront URLs)
- Remote state backend: S3 + DynamoDB (bootstrap script managed)
- Private RDS in private subnets; backend accesses via VPC connector
- NAT Gateway for backend outbound internet (required for Clerk, third-party APIs)
- Clerk issuer/audience optional but recommended for strict JWT validation

UPDATE: May 26–27, 2026 — Reprovision and Deploy Hardening

Context:
- Full dev destroy/reprovision cycle was executed with a new `cornerstone-dev` Clerk configuration.
- Terraform provision reached App Runner `CREATE_FAILED` after infrastructure creation.
- Deploy workflow later timed out while waiting for App Runner despite rollback already completing.

Findings:
- App Runner create failed when ECR repository existed but had no image (`latest` tag missing).
- Backend startup failed during deploy rollback due to stale `cornerstone-database-url` secret content and missing SSL mode.
- Smoke test failure at deploy end was caused by missing `access-control-allow-origin` for the CloudFront origin.
- Deploy wait loop treated `FAILED` and `ROLLBACK_FAILED` as terminal but not `ROLLBACK_SUCCEEDED`, causing false timeout messaging.

Implemented fixes:
- `provision-dev.yml`
  - Added early ECR bootstrap and placeholder image push so App Runner can pull an initial image at service creation time.
  - Added DB bootstrap/sync step that rewrites `RDS_DATABASE_URL_ARN` from the live RDS-managed master secret and endpoint.
  - Synced URL format to `postgresql+psycopg://...?...sslmode=require` for App Runner runtime compatibility.

- `deploy-dev.yml`
  - Updated App Runner wait logic to treat `ROLLBACK_SUCCEEDED` as terminal failure state.
  - Added CORS parameter reconciliation step to ensure current CloudFront origin is present in `/cornerstone-dev/cors_origins` before backend redeploy.

Operational result:
- App Runner service recovered to `RUNNING` after DB secret correction and redeploy.
- CORS header verification passed for CloudFront origin.
- Deploy smoke test path now aligns with real App Runner terminal statuses and current frontend origin behavior.

✓ Updated README.md with top-level warning not to share a single Clerk app across environments.
