Date: 2026-06-05
Topic: /api/me invalid-or-expired bearer token mitigation

Issue observed
- After successful auth and email verification, some sessions reached Home and showed:
	"Invalid or expired bearer token" from `/api/me`.

Analysis
- Frontend and backend Clerk instance configuration matched (`VITE_CLERK_PUBLISHABLE_KEY`
	and `CLERK_JWKS_URL` aligned).
- Backend did not enforce issuer/audience, reducing config mismatch risk.
- Likely cause: token freshness/timing race during post-auth navigation.

Change made
- Updated Home profile loader to:
	1) request a fresh Clerk token with `getToken({ skipCache: true })`
	2) call `/api/me`
	3) on 401 bearer-token auth error, refresh token and retry once

Validation
- Frontend lint: pass
- Frontend build (tsc + vite): pass

Date: 2026-06-05
Topic: Dark/light mode not respected on public auth pages

Issue observed
- Sign-in and sign-up pages rendered in light mode regardless of system preference.
- Protected app shell correctly respected system/saved theme preference.

Root cause
- `useTheme()` was only called inside `Layout`, which wraps protected routes.
- Public routes (/sign-in, /sign-up, /sso-callback) rendered outside `Layout`, so
  `applyTheme()` was never invoked and the `dark` class was never set on <html>.

Fix
- Added `useTheme()` call to the root `App` component.
- Theme is now initialized for every route, including public auth pages.
- `Layout` retains its own call; the hook is idempotent so there is no conflict.

Validation
- Frontend lint: pass
- Frontend build (tsc + vite): pass

File updated
- apps/web/src/App.tsx
